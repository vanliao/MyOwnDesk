# MyOwnDesk 技术决策记录

## 项目概要

- 远程桌面客户端：部署在 Windows 11，实现屏幕捕获、视频编码、输入注入
- 中继服务器：部署在 Ubuntu 云服务器，实现设备发现、会话配对、数据转发
- 优先保证画质和响应速度，追求流畅的操作体验
- 适配双屏场景（被控端双屏时，控制端可选择查看任意一个屏幕）
- 不考虑文件传输和声音传输
- 3-5 个客户端，任一可远程到任一另一客户端

## 决策清单

### #1 传输协议：QUIC

**结论**：使用 QUIC 协议，基于 `quinn` crate。

**理由**：
- 0-RTT 握手机制，连接建立比 TCP+TLS 快
- 独立流复用，无队头阻塞——视频帧丢包不会卡住输入事件
- 连接迁移，Wi-Fi 切换不中断连接
- 内建 TLS 1.3 加密，无需额外加密层
- `quinn` 是纯 Rust 实现，生态最成熟

**备选**：TCP（`tokio`）——队头阻塞在弱网下会导致鼠标和画面一起卡住，体验差。

---

### #2 屏幕捕获：DXGI Desktop Duplication

**结论**：使用 Windows DXGI Desktop Duplication API，通过 `windows-rs` 调用。

**理由**：
- GPU 零拷贝捕获，拿到 D3D11 纹理直接喂编码器
- 支持显示器刷新率（60fps+）
- 天然支持多显示器（每屏一个 `IDXGIOutputDuplication` 实例）
- Windows 8+ 均支持，Win11 上 WDDM 3.0 优化最佳

**备选**：
- GDI BitBlt：CPU 拷贝，30fps 上限，淘汰
- WGC (Windows.Graphics.Capture)：Win32 调用别扭，帧率不稳定

---

### #3 视频编码：FFmpeg + H.264

**结论**：使用 FFmpeg 的 H.264 硬件编码器。

**理由**：
- NVENC / QSV / AMF 三家 GPU 硬件编码均有成熟支持
- 无独显时 Intel QSV、AMD VCE 核显编码器可正常使用
- 无 GPU 时可回退到软件编码
- H.264 兼容性最好，所有 GPU 都支持硬件编解码
- 低延迟参数（`zerolatency`）调得最透

**编码参数**：
- 码率控制：CBR
- 码率值：10-20 Mbps（1080P 60fps）
- GOP：60 帧（1 秒一个关键帧）
- Preset：`ultrafast`
- Tune：`zerolatency`
- Profile：`high`

---

### #4 FFmpeg 接入：`ffmpeg-next` + 打包 DLL

**结论**：通过 `ffmpeg-next` crate 调用 FFmpeg，DLL 打包进安装目录。

**理由**：
- Safe Rust 封装，编解码流程无需手写 FFI
- 调参方便（zerolatency 等一行配置）
- DLL 打包后用户无需额外安装 FFmpeg
- 控制端解码也用同一 crate，代码不分裂

---

### #5 Crate 架构：三 crate

**结论**：

```
MyOwnDesk/
├── myowndesk-protocol/    ← 共享协议定义
├── myowndesk-client/      ← 桌面客户端
└── myowndesk-relay/       ← 中继服务器
```

**理由**：
- 协议层独立：Protobuf 定义、帧封装、消息类型两边共享
- 修改协议不会导致客户端和服务器代码耦合
- 将来加新客户端类型（如移动端）只需实现 protocol

---

### #6 序列化：Protobuf + `prost`

**结论**：使用 Protocol Buffers 作为消息序列化格式，通过 `prost` 编译。

**理由**：
- Schema 即文档，`.proto` 文件是客户端和服务器共享的真相来源
- 向后兼容（字段增删安全）
- `bytes` 字段零拷贝，视频帧数据不重复序列化
- 比 JSON 体积小，比 bincode 有 schema

---

### #7 输入模拟：`windows-rs` SendInput

**结论**：直接调用 Windows `SendInput` API。

**理由**：
- 绝对坐标控制，不依赖当前鼠标位置
- 双屏坐标映射由应用层计算（简单的缩放乘法）
- 控制端和被控端分辨率不一致时，缩放公式统一处理
- `windows-rs` 官方 crate，可靠

---

### #8 GUI 框架：`egui`

**结论**：使用 `egui` 即时模式 GUI。

**理由**：
- 轻量（~1MB），启动毫秒级
- 每帧全量重绘，和视频 60fps 刷新节奏天然一致
- 自定义 painter 可接入 D3D11 设备直接贴纹理
- 对于"几个按钮 + 大块远程画面"的界面足够

---

### #9 视频帧渲染：D3D11 纹理贴图

**结论**：解码后的帧通过 D3D11 纹理渲染到 egui 窗口。

**理由**：
- FFmpeg CPU 软解 → `UpdateSubresource` 上传到 D3D11 纹理 → egui 自定义 painter 贴图
- 1080P 60fps 下一次 GPU→CPU→GPU 传输，对现代硬件影响极小
- 后续可优化为 DXVA2 硬解（纹理直接输出，零拷贝）
- 核显、独显均支持 D3D11

---

### #10 加密：预留接口，暂不实现

**结论**：当前不实现端到端加密，但预留扩展接口。

**Proto 预留字段**：
- `bytes encrypted_payload`：加密后的帧数据
- `bytes nonce`：ChaCha20 随机数
- `uint32 key_version`：密钥版本号

**Rust trait 预留**：
- `FrameCipher` trait（`encrypt` / `decrypt`）
- 当前使用 `NoOpCipher` 空实现（透传）
- 后续替换为 `ChaChaCipher` 时调用方无改动

**理由**：个人项目，自己部署服务器，QUIC TLS 1.3 传输加密已足够。E2E 加密防的是服务器被入侵的场景。预留接口确保以后加加密时不改架构。

---

### #11 控制端解码：`ffmpeg-next` 软解

**结论**：控制端使用 CPU 软件解码 H.264。

**理由**：
- 1080P 60fps 软解在现代 CPU 上占用约 15%（一个核），几乎无感
- 控制端不捕获屏幕、不编码视频，CPU 本来就在闲置
- 和编码端同一 crate、同一 DLL，统一管理
- 后续可优化为 DXVA2 硬解

---

### #12 中继架构：一体化模型 A

**结论**：中继服务器兼任设备发现 + 数据转发。

**架构**：
```
客户端 A → 中继 ← 客户端 B
         ├── Register（设备上线注册）
         ├── Pair（发起连接请求）
         └── Relay（配对的连接间双向转发）
```

**理由**：
- 3-5 个客户端场景下无需信令/中继分离
- 一个进程、一个端口、配置简单
- 预留打洞扩展点：中继已知道双方地址，后续加 `try_hole_punch()` 即可
- 打洞失败时中继转发作为可靠回退

---

### #13 显示规格：1080P 60fps，单屏当前，双屏可选

**结论**：
- 每个显示器独立 DXGI 捕获实例
- 每个显示器独立 H.264 编码流
- 控制端始终显示一个窗口（单屏）
- 被控端双屏时，控制端可选择查看屏 1 或屏 2
- 切屏通过控制消息切换被控端的编码源

---

### #14 认证：预共享密钥

**结论**：所有客户端共用一个预共享密钥。

**流程**：
1. 中继首次启动时自动生成 256 位随机密钥
2. 将密钥复制到每个客户端的配置文件
3. 客户端注册时发送 `HMAC(key, device_id)` 作为认证令牌

**理由**：
- 3-5 台个人机器无需账号系统
- 配置一次即可，后续无感
- 预留扩展点：后续可加设备间授权白名单

---

### #15 QUIC 数据通道设计

**结论**：视频帧走 datagram，控制消息和输入事件走 stream。

```
一个 QUIC 连接内：
  Stream（可靠、有序）:
    - 控制消息：Register、Pair、切屏、心跳
    - 键鼠事件：每次点击/移动
    - 关键帧请求：「丢包了，发个关键帧过来」

  Datagram（不可靠、无重传）:
    - H.264 视频帧：丢一帧不影响，下一帧 16ms 后到
```

**理由**：视频帧重传浪费带宽且增加延迟，datagram 丢帧时画面短暂闪烁但不阻塞后续帧。控制消息和输入事件必须可靠送达，走 stream。

---

### #16 断线重连：简单重连按钮

**结论**：第一版不做自动重连。

**行为**：
- 控制端检测到对端断开 → 显示"连接已断开"→ 用户手动点"重连"
- 客户端到中继的 QUIC 连接通过 idle timeout 检测心跳
- 后续可优化为指数退避自动重连

**理由**：自动重连边界条件多（谁先上线？旧会话怎么清理？），不值得第一版投入。

### #20 进程架构：单 exe 双角色

**结论**：一个 exe，两种启动模式。

```rust
fn main() {
    match std::env::args().nth(1).as_deref() {
        Some("--service") => run_service(),  // Windows 服务，Session 0，SYSTEM 权限
        _                 => run_gui(),       // 普通桌面，Session 1，用户权限
    }
}
```

**架构**：
```
客户端 (myowndesk-client.exe)
  │
  ├── 服务模式（--service，Windows 服务，开机自启）
  │     ├── 屏幕捕获 (DXGI)
  │     ├── 视频编码 (FFmpeg)
  │     ├── 输入注入 (SendInput)
  │     ├── 网络连接（到中继）
  │     └── 锁屏控制
  │
  └── GUI 模式（默认，双击启动）
        ├── 查看在线设备列表
        ├── 发起/断开连接
        └── 配置设定
```

**GUI ↔ 服务 IPC**：通过 `127.0.0.1` 上的本地 TCP 端口通信，`tokio` TCP 实现。

**理由**：
- Windows Session 0 隔离：服务进程在 Session 0 无法显示 GUI，GUI 进程在 Session 1 无权限访问锁屏桌面
- 单 exe 分发简单，用户通过启动参数区分角色
- 本地 TCP IPC 最简单可靠，无需额外依赖

### #21 锁屏功能：trait 隔离 + 分阶段实现

**结论**：通过 `InputBackend` trait 将锁屏相关功能隔离，先实现普通桌面模式，最后攻克服务模式。

**trait 设计**：

```rust
pub trait InputBackend: Send + Sync {
    fn send_key(&self, key: KeyEvent) -> Result<()>;
    fn send_mouse_move(&self, x: i32, y: i32) -> Result<()>;
    fn send_mouse_click(&self, button: MouseButton, down: bool) -> Result<()>;
    fn lock_screen(&self) -> Result<()>;
}

// 实现 1：普通桌面模式（先做）
pub struct DesktopInput;

// 实现 2：服务模式（后做，可失败）
pub struct ServiceInput;
```

**实现优先级**：
1. `DesktopInput` — 先跑通全部核心功能（桌面捕获、编码、传输）
2. `ServiceInput` — 最后攻克（锁屏下输入 + 断开后锁屏）

**模块影响分析**：

| 模块                     | 锁屏是否影响 | 说明                 |
| ---------------------- | ------ | ------------------ |
| DXGI 屏幕捕获              | ❌ 不影响  | 捕获显存画面，锁屏/桌面同一张图   |
| FFmpeg 编码              | ❌ 不影响  | 拿到帧就编码，不管帧内容       |
| QUIC 传输                | ❌ 不影响  | 字节流，不理解上层内容        |
| egui GUI               | ❌ 不影响  | 运行在控制端，和被控端状态无关    |
| 设备发现/配对                | ❌ 不影响  | 中继协议，不管对端桌面状态      |
| 输入注入 (SendInput)       | ✅ 影响   | 锁屏下输入是实现重点         |
| 锁屏控制 (LockWorkStation) | ✅ 影响   | 断开后通过 WTS API 触发锁屏 |

**失败回退**：如果服务模式下的锁屏输入做不出来，不影响普通桌面模式的全部功能。挂锁屏幕画面仍然可见，仅输入被安全桌面拦截。

**理由**：锁屏相关是 Windows 安全架构导致的技术难点，隔离后不影响核心流程的开发和验证。
